# L1 · Pattern（组合构式层）

## 目的

通过全文阅读理解，识别那些**不是由单个词项独立生效**、而是由多个词、多个子句、前后顺序、许可关系或外部引用共同形成的 Steering 组合构式。

- **L1-Pattern（本文档）**：学习/分析流程，描述"如何从材料中发现构式候选、进行语义验证、再决定是否沉淀为规范"
- **`assets/lang/L1-Pattern.md`（待建立）**：学习成果，积累已通过验证的 Gosh 组合构式规范
- **`L2-Grammar-Pass.md`**：检查文档是否符合已经沉淀的语法规则，不负责发现新的组合构式

当前仓库中尚未建立 `assets/lang/L1-Pattern.md`。在该规范文件建立前，本文档先承担**研究方法**与**模式候选归档**的职责；已通过验证的 Pattern，后续可根据成熟度决定：

- 独立沉淀为 `assets/lang/L1-Pattern.md`
- 或转写为 `assets/lang/L2-Grammar.md` 的稳定规则来源

L1-Pattern 不直接宣布某个组合"已经有效"。它负责给出候选发现方法、语义验证问题、分层边界与迁移门槛。

---

## 为什么单列 Pattern 层

在单个词项研究与既有规则合规性检查之间，真实材料中存在一批现象，它们的 Steering 效果显然**不只来自单个词**：

- `NEVER X, instead Y`：效果来自"禁止 + 替代"的组合，而不只来自 `NEVER`
- `IMPORTANT -> MUST -> 行为细节 -> MUST`：效果来自升级链，而不只来自某个单句
- `Only update requirement.txt, never rewrite`：效果来自同一句中的双力对冲与收束
- `without explicit user permission` vs `unless explicitly asked`：效果来自许可结构类型差异，而不只是某个词更强
- `MUST be structured exactly as follows` + 后附模板：效果来自句内约束与句外规范的联动

因此需要单列 Pattern 层，专门处理：**关系发力、组合发力、构式发力**。

---

## 分层边界

### 不属于本文档的现象

以下问题不应作为 Pattern 候选立案：

- 单个令词是否具有强候选价值（如 `MUST`, `NEVER`, `ALWAYS`, `CRITICAL`）
- 裸动词祈使是否应视为非标准 Imperative 候选（如 `MINIMIZE`, `MARK`, `RESOLVE`）
- 单个词项的纯形式显著性（全大写、独立出现）
- 局部精度增强但仍以单句单词为核心的情形

### 属于本文档的现象

以下问题由本文档处理：

- 一个模式的 Steering 是否来自多个成分的关系，而非单词本体
- 一个句型是否通过"禁止 + 替代"、"义务 + 禁止"、"门控 + 例外"等组合形成新的效果
- 一条约束是否依赖多句链路、外部引用、结构位置或规则联动才能成立
- 一种模式是否可抽象为稳定可迁移的构式模板

---

## 触发条件

文档中出现以下任一情况时，执行 L1-Pattern：

- 两个以上子句共同施加约束，且去掉其中一部分后效果显著变化
- 约束效果依赖"禁止后给替代"而非单独禁止
- 约束效果依赖升级链、重复链、首尾呼应等跨句组织
- 约束效果依赖许可结构（门控条件 / 例外子句 / 授权边界）
- 约束效果依赖引用外部模板、schema、example、流程块
- 同一个模式在多个 Agent 中重复出现，值得抽象为模板

---

## 候选评分（研究优先级，不是有效性结论）

使用 5 分制评估每条 Pattern 作为研究候选的优先级。该分值只回答"这个组合模式值不值得验证"，不直接回答"该模式是否已经被证明有效"。

| 分值 | 判定条件 | 处理方式 | 特征示例 |
|------|----------|----------|----------|
| 5 | 关系明确 + 变量边界清晰 + 易于构造对照实验 + 跨 Agent 可复现 | 最高优先级进入语义验证 | `NEVER refer to tool names, instead say what the tool is doing.` |
| 4 | 组合关系清晰，且看起来明显影响执行边界或执行路径 | 优先进入语义验证 | `Only update requirement.txt, never rewrite.` |
| 3 | 有组合价值，但宿主句依赖较强或边界不够稳定 | 视主题决定是否验证 | `IMPORTANT. MUST do X. MUST do Y.` |
| 2 | 看起来像模式，但更可能只是排版或个别写法 | 保留为对照样本 | 单次出现、无法稳定抽象的格式化链 |
| 1 | 形式上有组合，语义上不形成可复用约束 | 保留为反例池 | 纯修辞性并列或说明性转折 |

**4-5 分模式优先进入语义验证；1-2 分模式不要直接丢弃，应保留为反例或对照样本。**

## 语义验证（必须使用大模型语义思考）

候选分不能证明组合模式已经有效。对每个值得实验的 Pattern，必须让大模型比较：**模式存在时与模式被拆解/替换时，Steering 是否发生可解释变化。**

### 验证步骤

1. **固定命题，只替换模式关系**：尽量不改变任务内容，只改变组合方式。
2. **构造 2-4 个对照版本**：至少包含"保留原模式 / 拆掉一部分 / 改成近义结构"。
3. **要求大模型解释差异来源**：不是只问哪个更强，而是问为什么更强。
4. **交叉验证语料证据**：只有当模型判断与真实语料复现相互支撑时，才允许沉淀。

### 推荐提问方式

#### 示例 1：禁止 + 替代

```text
A: NEVER refer to tool names, instead say what the tool is doing.
B: NEVER refer to tool names.
C: Say what the tool is doing instead of referring to tool names.

这三个版本在约束强度、替代路径清晰度、自我执行性上有什么差异？
真正起作用的是 NEVER，还是“instead Y”给出的替代路径？
```

#### 示例 2：门控条件 vs 例外子句

```text
A: Do not install dependencies without explicit user permission.
B: Do not install dependencies unless explicitly asked.

这两个结构在授权来源、Agent 自我判断空间、违规风险上有什么不同？
```

#### 示例 3：双力句

```text
A: Only update requirement.txt, never rewrite.
B: Only update requirement.txt.
C: Never rewrite requirement.txt.

双力组合是否比单边约束更容易形成边界？
```

### 记录维度

至少覆盖以下问题：

- 去掉其中一个子句后，约束是否明显变弱
- 替代路径、许可边界、执行顺序是否更清晰
- Agent 是否更容易自我执行、自我检查、自我纠偏
- 模式效果是否依赖位置、标题、列表、模板引用等额外结构
- 模式是否可稳定迁移到其他命题，而不仅仅对单个例句有效

记录格式：`样本ID | 模式类型 | 对照改写 | 模型判断摘要 | 暂定结论 | 是否建议沉淀`

## 核心模式类型

### P1：禁止 + 替代（Prohibition + Replacement）

**定义**：先显式禁止旧行为，再给出替代行为路径。

**原型**：

- `NEVER X, instead Y`
- `Do not X; say Y instead`

**研究重点**：

- 替代子句是否只是补充说明，还是实际承担纠偏作用
- 去掉替代子句后，Agent 是否更容易停在"知道不能做，但不知道该怎么做"

### P2：多句约束升级链（Escalation Chain）

**定义**：同一约束跨多句不断被强化、细化或重复。

**原型**：

- `IMPORTANT -> MUST -> 行为细节 -> MUST`
- 标签句 + 裸祈使句 + 例外/范围句

**研究重点**：

- 升级链是增强注意力，还是制造噪音
- 重复是否带来真正的执行稳定性，而不只是视觉强度

### P3：双力句（Dual-Force Constraint）

**定义**：同一句中同时存在义务与禁止、收窄与排除、主路径与禁区。

**原型**：

- `Only update requirement.txt, never rewrite`
- `You MUST verify first and NEVER skip diagnostics`

**研究重点**：

- 双力是否比单力更清晰
- 哪一半承担主边界，哪一半负责封口

### P4：元约束（Meta-Constraint）

**定义**：不是直接约束动作，而是约束"如何理解和执行其他约束"。

**原型**：

- `You MUST treat each constraint as a strict requirement`
- `Interpret every prohibition as absolute unless otherwise stated`

**研究重点**：

- 它是否对下游约束具有稳定放大作用
- 它的效力是否依赖后文已有强约束网络

### P5：许可边界构式（Gate / Exception Pattern）

**定义**：通过许可来源、例外条件、外部授权来划定边界。

**原型**：

- `unless explicitly asked`
- `without explicit user permission`
- `only if the user confirms`

**研究重点**：

- 是 Agent 自行判断，还是必须等待外部许可
- 许可结构差异是否真实改变风险控制

### P6：外部引用锚定构式（External Reference Anchoring）

**定义**：句内约束通过模板、schema、example、checklist 等外部对象获得精度。

**原型**：

- `MUST be structured exactly as follows` + 后附模板
- `Follow the schema below`
- `Use the example format shown here`

**研究重点**：

- 精度是否主要来自句内词项，还是来自被引用对象
- 外部锚点是否让约束更可执行、更可检查

## 识别维度

### 维度 1：全文阅读理解

完整阅读目标文档，先找出"哪些地方不是单个词在发力，而是关系在发力"。

**识别步骤**：

1. 定位包含多个约束子句、多个连接关系、多个授权信号的片段。
2. 判断其中的 Steering 是否来自单词本体，还是来自组合关系。
3. 若主要来自组合关系，则标记为 Pattern 候选，而非单纯 Lexicon 样本。

记录格式：`候选片段 | 所在位置 | Agent | 初步模式类型 | 初步判断`

### 维度 2：模式拆解

把候选模式拆成最小作用部件，明确每个部件承担什么功能。

  **识别步骤**：
  
  1. 拆出主约束子句、替代子句、门控子句、引用子句、重复强化子句。
  2. 记录结构信号：是否有前置标记、视觉增强、叠加强度、复合结构、例外子句、范围限定。
  3. 标记每个部件的功能：禁止 / 替代 / 授权 / 顺序 / 精度 / 放大。
  4. 判断如果删掉其中一个部件，模式是否还成立。
  
  记录格式：`样本ID | 子句拆解 | 结构信号 | 部件功能 | 缺失影响`

### 维度 3：语义验证

对 4-5 分候选进行模式级对照验证。核心问题不是"它像不像某种规则"，而是"这个组合关系是否真的让 Agent 更好执行"。

**验证步骤**：

1. 固定命题，生成去模式化版本与近义模式版本。
2. 向大模型提问差异来源。
3. 记录通过 / 存疑 / 不通过。

记录格式：`样本ID | 对照改写 | 模型判断摘要 | 结论 | 备注`

### 维度 4：跨 Agent 聚类

将相同模式在不同 Agent 中做横向对比，判断它是单例写法，还是具有跨语料复现价值。

**归纳步骤**：

1. 按模式类型分组（P1-P6）。
2. 比较同类模式在不同 Agent 中的宿主词项、位置、验证结果。
3. 只有在语义验证与跨 Agent 复现相互支撑时，才提炼可迁移模板。

记录格式：`模式类型 | 样本数量 | Agent 分布 | 验证结论 | 可迁移模板`

## 边界样例

更适合作为 Pattern 候选观察的现象：

- **EC2：多句约束升级链**
- **EC4：禁止 + 替换模式**
- **EC6：义务 + 禁止双力句**
- **EC7：元约束（Meta-Imperative）**
- **EC8：门控条件 vs 例外子句**
- **EC10：外部引用精度**

更适合作为非构式样本保留观察的现象：

- **EC1：全大写非标准祈使动词**
  - 本质仍是"非标准令词候选如何记分"
- **EC3：结构化前缀（非关键词）**
  - 更像形式承载与注意力标记，可暂作为显著形式特征
- **EC5：小写形式 + 高精度**
  - 仍以单句单词候选为中心，只是精度机制补偿形式弱点
- **EC9：MUST 饱和度（密度稀释）**
  - 更像分布统计与整段注意力竞争，暂不直接归为 Pattern

## 我的当前判断

1. **Pattern 不是 Grammar 的别名。**
  - Grammar 更偏"已有规则是否合规"
  - Pattern 更偏"尚未沉淀的新组合现象如何被发现、验证、归纳"

2. **Pattern 不是单词候选研究的附录。**
  - 当研究对象已经从单词本体转移到"子句之间的功能关系"时，就应独立立案

3. **`NEVER X, instead Y` 应视为 Pattern 层原型，不应再作为单纯令词边缘条目处理。**
  - 这里真正值得研究的是"禁止 + 替代"构式，而不是 `NEVER` 本身又多了一分

4. **L1-Pattern 是一个必要的缓冲层。**
  - 它避免把所有新观察都直接硬塞进 Grammar
  - 也避免构式现象长期悬空，没有稳定的发现入口

5. **未来若 Pattern 层稳定，应新增对应 assets 文档。**
  - 否则发现层与规范层会长期失衡：能分析，但无规范落点
