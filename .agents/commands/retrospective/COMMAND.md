# Retrospective 复盘

这是一个轻量复盘能力，继承 Skill Evolution 的关键处理方式：复盘诊断、经验提取、缝合对齐。

默认仅输出本次会话复盘，不自动写入任何文件。

## 核心职责

1. **复盘诊断 (Session Review)**：在对话结束时，分析本次会话全流程中的目标、决策、问题与结果。
2. **经验提取 (Experience Extraction)**：从用户反馈与执行过程中提炼可复用经验（偏好、修复点、提示词策略等）。
3. **智能缝合 (Smart Stitching)**：将经验整理为可沉淀的 Markdown 条目，并在落盘前先征求用户确认。

## Workflow

### 0. 输入类型识别（Input Type Detection）

在开始复盘前，Agent 必先识别输入类型，选择合适的复盘策略。

**输入类型分类**:
1. **完整对话内容** - 包含用户请求、Agent回复、工具调用、结果反馈
   - 适用: 快速复盘 / 深度复盘
   - 信息完整度: 高
   
2. **执行日志（元数据类）** - 包含 task_metadata.json, focus_chain.md
   - 适用: 元数据复盘模板
   - 信息完整度: 中（缺少对话细节）
   
3. **聚焦链驱动任务** - 包含明确的 focus_chain 清单
   - 适用: 聚焦链复盘模板
   - 信息完整度: 中（有任务步骤，缺少决策过程）
   
4. **API对话历史** - 包含 api_conversation_history.json, ui_messages.json
   - 适用: 快速复盘（提取关键节点）
   - 信息完整度: 中高（有对话，但可能缺少上下文）

**识别流程**:
1. 检查输入文件类型和内容
2. 判断信息完整度（完整对话 vs 元数据 vs 聚焦链）
3. 选择合适的复盘模板和策略
4. 如果信息不完整，提示用户或跳过相关部分

**信息缺失处理**:
- 缺少对话细节 → 跳过"用户澄清点"部分，仅复盘任务执行
- 缺少问题解决过程 → 仅记录最终解法，跳过"尝试路径"
- 缺少用户反馈 → 仅提取隐含信号（任务完成状态、上传结果）

### 1. 经验复盘 (Review)
当用户触发复盘时，Agent 必须执行：
1. **扫描上下文**：定位本次会话中的关键节点（需求澄清、方案选择、改动落地、结果反馈）。
2. **识别信号**：提取满意点与不满意点（报错、风格偏差、参数不合预期、沟通成本高等）。
3. **归因总结**：明确"为何有效 / 为何失效"，避免只记录现象。

#### 引导性问题（Review Checklist）
在复盘过程中，Agent 应依次回答以下问题，确保覆盖关键维度。

支持两种复盘模式：
- **快速复盘**：仅回答必须问题（标记 ⚡），输出精简模板（约 100-200 行）
- **深度复盘**：回答所有问题，输出完整模板（约 500+ 行）

用户可选择模式，默认为快速复盘。

---

**⚡ 目标与结果（必须回答）**
- 本次会话的核心目标是什么？
- 目标是否完全达成？部分达成？还是失败？
- 结果与预期的差距在哪里？

**⚡ 关键决策（必须回答）**
- 做了哪些关键决策？（至少识别3个）
- 每个决策有哪些候选方案？
- 为什么选择最终方案？是否有更好的替代方案？
- 决策带来的结果如何？是否有意外影响？

**⚡ 问题与解决（必须回答）**
- 遇到了哪些主要问题？（技术问题、理解偏差、沟通成本）
- 尝试了哪些解决路径？哪些成功？哪些失败？
- 最终采用的解法是什么？为什么有效？

**⚡ 用户反馈（必须回答）**
- 用户明确表达了哪些偏好？（喜欢的方式、不喜欢的方式）
- 用户对结果满意度如何？（明确反馈或隐含信号）

---

**效率与成本（可选回答）**
- 哪些步骤耗时最长？是否有优化空间？
- 是否有不必要的来回沟通？
- 是否有可以提前准备的上下文或知识？

**可复用经验（可选回答）**
- 哪些做法值得固化？（模板、流程、提示词）
- 哪些错误值得记录在避坑指南？
- 哪些偏好值得写入用户画像？

**深度归因分析（可选回答，仅深度模式）**
- 每个决策是否有更深层的归因？（团队约束、时间压力、技术限制）
- 每个问题是否有系统性原因？（流程缺陷、工具不足、知识缺失）
- 是否有隐藏的依赖关系或连锁反应？

### 2. 经验提取 (Extract)
将非结构化反馈转成 Markdown 经验条目，建议包含：
- 用户偏好（Preferences）
- 修复与规避（Fixes）
- 可复用提示（Prompts / Playbook）
- **决策追溯（Decision Trace）**：关键决策点、选择理由、替代方案及影响
- **问题解决路径（Problem Solving Path）**：遇到的问题、尝试的方案、最终解法、学到的教训

### 3. 缝合建议 (Stitch Preview)
在对话内给出"可沉淀版本"的 Markdown 建议稿（不自动写文件），例如建议沉淀到：
- 指定 Skill 的 `SKILL.md`
- 相关 `references/*.md`

#### 沉淀目标识别
根据经验类型，推荐沉淀位置：
- **用户偏好** → `.agents/user-preferences.md` 或项目级 `docs/user-profile.md`
- **工具技巧** → 相关 Skill 的 `references/usage-tips.md`
- **最佳实践** → 相关 Skill 的 `references/best-practices.md`
- **避坑指南** → 相关 Skill 的 `references/pitfalls.md`
- **决策模式** → `.agents/commands/decision-log.md`
- **问题解决** → `.agents/commands/problem-solutions.md`

#### 沉淀路径建议
提供具体沉淀路径，例如：
```markdown
建议沉淀路径：
1. 决策追溯 → `.agents/commands/decision-log.md` (新增 "技术选型" 章节)
2. 问题解决 → `.agents/skills/[相关skill]/references/pitfalls.md` (补充到现有章节)
3. 用户偏好 → `.agents/user-preferences.md` (更新 "代码风格偏好" 条目)
```

### 4. 询问是否沉淀 (Confirm)
复盘输出完成后，必须追加确认：
- “以上是本次会话复盘，是否需要我将这些经验沉淀到指定文档？”

> 未获得用户明确同意前，不执行任何写入沉淀动作。

## 输出格式

复盘只用 Markdown，建议结构：
- 背景与目标
- 过程回顾
- 做得好的点
- 可以改进的点
- 可沉淀经验草案
- 是否沉淀确认

### 结构化模板

支持两种模板：
- **快速复盘模板**：精简版，聚焦核心洞察（约 100-200 行）
- **深度复盘模板**：完整版，包含详细分析（约 500+ 行）

## 最佳实践

- 先总结事实，再给判断，避免主观泛化。
- 重点记录"有效做法 + 可避免的问题"。
- 保留原版处理方式，但以轻量对话形态执行，不依赖脚本与 JSON。

### 复盘质量自检

在输出复盘结果前，Agent 应自检以下质量指标。

支持两种质量标准：
- **快速复盘质量标准**：聚焦核心要素，输出精简
- **深度复盘质量标准**：覆盖全面要素，输出详细

---

#### 快速复盘质量标准

**完整性检查（必须）**
- ✓ 是否识别了至少 3 个关键决策？
- ✓ 是否识别了至少 3 个主要问题？
- ✓ 是否提取了用户偏好？
- ✓ 是否提供了至少 3 个可沉淀经验？

**深度检查（必须）**
- ✓ 每个决策是否说明了选择理由和影响？
- ✓ 每个问题是否说明了最终解法和教训？
- ✓ 是否避免了只记录现象？

**实用性检查（必须）**
- ✓ 沉淀建议是否明确了目标路径？
- ✓ 输出是否精简（100-200 行）？
- ✓ 是否在最后询问了沉淀确认？

**输出格式检查（必须）**
- ✓ 是否使用了快速复盘模板？
- ✓ Markdown 格式是否清晰？
- ✓ 是否聚焦核心洞察，省略次要细节？

---

#### 深度复盘质量标准

**完整性检查（必须）**
- ✓ 是否识别了至少 4 个关键决策？
- ✓ 是否识别了至少 4 个主要问题及解决路径？
- ✓ 是否提取了用户偏好（明确表达和隐含信号）？
- ✓ 是否提供了至少 6 个可沉淀经验（涵盖所有类别）？

**深度检查（必须）**
- ✓ 每个决策是否说明了候选方案、选择理由、影响、替代方案分析？
