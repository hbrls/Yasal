# Learn - 从AI交互历史中学习

## 核心理解

**学习过程本质**：不是简单的文本分析，而是从AI与用户交互的元数据中挖掘行为模式，量化识别问题，沉淀结构化教训。

## 学习材料的构成

### 原始材料（放入 `lessons/` 目录）

学习材料可以是**任何格式的文本**：JSON、CSV、Markdown、纯文本日志、二进制日志等。关键不是格式，而是从中**提取通用的行为信号**。

### 通用行为信号（不依赖格式）

Instead of assuming specific fields, we extract:

1. **时间戳信号**
   - 任何时间记录（日期、时刻、纪元秒）都可以识别
   - 用于检测：编辑间隔、操作频率、修正循环

2. **重复模式**
   - 同一实体（文件名、任务ID、字符串）多次出现
   - 用于检测：重复犯错、反复修正、循环编辑

3. **修正轨迹**
   - 同一内容被不同角色（AI/用户）修改
   - 用于检测：编辑冲突、验证失败、需要澄清

4. **操作序列**
   - 事件或操作的先后顺序
   - 用于检测：一次性完成 vs. 迭代修复、依赖关系错误

5. **量化指标**
   - 任何可数的属性（行数、字符数、操作数、出现频次）
   - 用于检测：规模突变、异常加重

## 学习工作流

### Phase 1: 扫描与解析材料
- 读取 lessons/ 目录下所有文件
- **格式自适应解析**：不假设结构，自动识别时间戳、重复出现的键、修改轨迹等通用特征
- 建立统一的内部表示：时间序列、实体-操作关系图、角色变化序列

### Phase 2: 量化问题检测
从**通用行为信号**提取问题模式，而非主观判断：

| 检测项 | 通用方法 | 阈值 |
|--------|----------|------|
| 重复修正 | 同一实体在材料中出现修改 ≥3 次，且修改都来自不同角色 | ≥3次修改向不同角色 |
| 快速循环 | 同一实体有连续修改，时间戳间隔很小（通常秒级或分钟级） | 间隔<60s且循环≥2次 |
| 高频操作 | 同一类型操作（如编辑、读取、删除）的出现次数 | >均值的2倍标准差 |
| 不完整状态 | 操作序列中断或存在悬停状态（有开始无结束） | 悬停数量>10% |
| 资源不一致 | 同一资源被引用或修改时用不同的标识符 | 相同资源用>2种标识 |

### Phase 3: 提取结构化教训
每条教训必须包含：
- **Title**: 问题描述（简洁有力）
- **Category**: problem / win / meta-insight
- **Outcome**: 观察到的结果（带量化数据）
- **Insight**: 可执行的改进策略（带触发条件和阈值）
- **Source**: 来源文件路径+字段引用（可追溯）

### Phase 4: 质量评分迭代
运行 `scripts/quality_score.py` 计算三维度分数：
- **可操作性**: Outcome是否有明确触发信号和数据
- **可追溯性**: Source是否引用具体字段
- **指导价值**: Insight是否包含阈值、步骤、策略关键词

目标分数：80+
低于目标时迭代优化Insight的可执行性。

## 与codex-autoresearch的关系

Learn命令本身遵循autoresearch原则：

1. **机械Metric**: 质量分数（0-100），而非主观"看起来好"
2. **One Change Per Iteration**: 每轮改进一个教训的Insight或评分算法
3. **Keep/Discard Decision**: 分数提升→keep，分数下降→discard
4. **Pivot机制**: 卡住时切换策略（从优化教训内容→优化COMMAND.md本身）

## 实际使用场景

### 场景1: 发现AI重复犯错（格式自适应）
- 用户注意到某个任务反复失败、某个文件被修改多次
- 将交互历史放入lessons/（任何格式都可以）
- 运行learn命令，自动检测"同一实体多次修改"
- 输出教训：哪些实体有高频修正，暗示AI理解不足或需求不清

### 场景2: 分析操作序列质量
- 用户想对比两次任务执行：一次一遍过，一次反复修改
- 放入两份交互历史（JSON、CSV、纯文本都可以）
- learn命令统计：速度（时间戳间隔）、循环次数、涉及资源数
- 输出win类教训：什么条件下一遍过？一遍过的关键特征是什么？

### 场景3: 跨会话知识积累（格式无关）
- 每次任务后保存交互历史（任何格式）
- 定期运行learn，积累lessons文件
- 框架自动从各种格式中提取通用规律
- 整合出的教训对未来任何格式的交互都有指导

## 核心价值

1. **从错误中学习**: AI系统不再盲目重复犯错
2. **量化驱动**: 基于数据而非主观判断识别问题
3. **知识沉淀**: 教训文件成为可复用的资产
4. **持续改进**: 质量评分机制确保教训质量不断提升

## 核心设计原则

| 原则 | 说明 |
|------|------|
| **格式无关性** | 不假设输入的结构。任何包含时间戳、重复、序列的文本流都能处理 |
| **信号驱动** | 从通用行为信号（重复、循环、频率、不一致）推导问题，而非依赖字段名 |
| **泛化优于拟合** | 目标是发现跨格式、跨项目都适用的学习规律，而非解析特定JSON结构 |
| **量化而非主观** | 每个检测都有明确的数值阈值和统计方法，可重复验证 |

## 未来演进方向

1. **自动触发**: 任务完成后自动保存metadata并触发learn
2. **跨项目共享**: lessons库可跨项目引用，形成通用AI行为知识库
3. **实时预警**: 任务执行中实时监控user_edited频率，达到阈值即时警告
4. **教训应用**: 将提取的Insight直接嵌入AI执行策略，而非仅作为参考文档

---

**核心洞察**: Learn是AI的**自我进化机制**，通过从交互历史中提取**通用行为规律**而非特定格式规则。它应该能处理任何形式的交互记录——JSON、CSV、纯文本、数据库日志——只要里面有时间戳、操作序列、修改轨迹，就能挖掘出对所有未来交互都适用的学习规律。这样的框架才不会被某一次的学习材料"训练坏"。
